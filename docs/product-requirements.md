# 在线 COC 跑团助手产品需求

## 1. 总目标

开发一个面向 COC 跑团的在线助手，支持 KP 与玩家在线协作完成跑团准备、游戏过程、规则查询、投掷记录、角色卡管理、录音与会后总结。

最终版本必须覆盖：

- 在线文字聊天。
- 在线语音或录音交流。
- 跑团过程录音与结束总结。
- 常见 COC 骰子与快捷检定。
- 上传角色卡后自动填充属性、技能、武器、装备、背景故事与调查员经历。
- KP 可查看、修改、备注玩家角色卡。
- 玩家可基于自己的角色卡进行一键投掷。
- 房间内实时显示玩家关键属性、技能与状态。
- 快速查看 COC 规则书。
- 规则书查询必须保证准确性，有本地资料来源和页码引用。
- 支持自定义界面背景，默认纯黑，并提供其他纯色界面选项。

## 2. 用户角色

### KP

KP 是房间主持者，拥有最高管理权限。

能力：

- 创建和关闭跑团房间。
- 邀请玩家进入房间。
- 查看所有玩家角色卡。
- 修改、备注、锁定角色卡字段。
- 发起公开或私密投掷。
- 查看所有骰子日志。
- 管理规则书资料。
- 开始、暂停、结束录音。
- 生成跑团总结。

### 玩家

玩家参与房间并控制自己的角色。

能力：

- 加入房间。
- 上传自己的角色卡。
- 查看自己的角色卡。
- 按角色卡属性/技能快捷投掷。
- 发送文字消息。
- 发送语音留言或参与语音。
- 查看公开规则查询结果。
- 查看公开投掷日志。

## 3. 功能需求

### 3.1 房间与成员

必须支持：

- 创建跑团房间。
- 通过邀请码或链接加入房间。
- 区分 KP 与玩家身份。
- 显示成员在线状态。
- 显示每个玩家绑定的角色。
- 房间状态包括：准备中、进行中、已结束。

后续增强：

- 房间密码。
- 旁观者身份。
- 多 KP。
- 跑团战役归档。

验收标准：

- KP 可以创建房间并让玩家加入。
- 房间内可以看到成员列表。
- 系统能正确识别 KP 和玩家权限。

### 3.2 文字聊天

必须支持：

- 房间内实时文字聊天。
- 系统消息。
- 骰子结果消息。
- 角色卡更新消息。
- 规则引用消息。

后续增强：

- 私聊。
- KP 私密频道。
- 消息搜索。
- 消息引用。

验收标准：

- 任意成员发送消息后，房间内其他成员能实时看到。
- 骰子结果以不可伪造的系统消息出现。
- 聊天记录能在刷新后保留。

### 3.3 语音、录音与总结

第一阶段必须支持：

- 浏览器录音。
- 上传录音为语音消息。
- 在聊天中播放录音。
- 跑团结束后基于文字聊天和录音转写生成总结草稿。

最终版本必须支持：

- 多人实时语音房间。
- 服务端录制。
- 自动语音转写。
- 跑团结束后一键总结。

总结内容包括：

- 本次剧情概要。
- 关键线索。
- 重要 NPC。
- 玩家行动。
- 角色状态变化。
- 关键骰子结果。
- 未解决问题。
- 下一次跑团建议。

验收标准：

- 玩家或 KP 能录制并发送语音。
- 录音文件能播放。
- 结束房间后能生成一份可编辑总结。

### 3.4 骰子与检定

必须支持：

- 基础骰子表达式：`1d100`、`1d6`、`2d6+3`。
- COC 检定：普通、困难、极难。
- 大成功与大失败判定。
- 奖励骰与惩罚骰。
- 属性快捷检定。
- 技能快捷检定。
- 暗骰。
- 投掷日志。

原则：

- 骰子结果必须由后端生成。
- 玩家不能本地伪造投掷结果。
- 投掷结果必须记录操作者、时间、表达式、目标值、难度和结果。

验收标准：

- 玩家点击角色卡属性后可以直接投掷。
- 投掷结果能显示成功等级。
- 所有公开投掷同步到聊天。
- KP 可以查看完整投掷历史。

### 3.5 角色卡上传与自动解析

必须支持：

- 上传 Excel 角色卡。
- 解析基础信息：姓名、玩家、职业、年龄、性别、时代、住地、故乡。
- 解析属性：STR、DEX、POW、CON、APP、EDU、SIZ、INT、Luck。
- 自动计算困难值和极难值。
- 解析衍生状态：HP、SAN、MP、MOV、DB、Build、Armor。
- 解析技能表。
- 解析武器表。
- 解析资产与随身物品。
- 解析背景故事、形象描述、思想与信念、重要之人、意义非凡之地、宝贵之物、特质、难言之隐、伤口和疤痕、恐惧症和躁狂症。
- 解析调查员经历。
- 解析神话相关记录与法术。

KP 管理能力：

- 查看所有角色卡。
- 修改角色卡字段。
- 增加 KP 备注。
- 锁定关键字段。
- 查看角色卡修改历史。

玩家能力：

- 查看自己的角色卡。
- 发起自己角色卡上的属性/技能投掷。
- 提交角色卡更新请求。

验收标准：

- 上传当前工作区内的 COC Excel 模板后，系统能识别主要字段。
- 解析结果能展示在网页角色卡面板中。
- KP 修改后，玩家侧能看到更新。
- 角色卡字段可以被用于快捷投掷。

### 3.6 规则书检索

必须支持：

- 导入本地规则书 PDF。
- 按页码建立索引。
- 关键词检索。
- 展示结果片段和页码。
- 从规则查询结果发送引用到聊天。

准确性要求：

- 不能凭模型记忆回答规则。
- 不能无来源回答。
- 查询结果必须显示规则书来源和页码。
- 如果没有找到可靠结果，必须明确提示没有找到。

后续增强：

- 语义搜索。
- 多规则书管理。
- 规则问答。
- 常用规则收藏。
- KP 自定义规则。

验收标准：

- 搜索关键词后能返回相关页码。
- 每个结果都能看到来源文件和页码。
- 系统不会在无检索结果时编造规则。

### 3.7 KP 控制台

必须支持：

- 房间成员列表。
- 角色卡总览。
- 玩家状态总览。
- 投掷日志。
- 规则查询。
- 总结生成入口。

后续增强：

- 暗骰面板。
- 私密线索投递。
- NPC 管理。
- 手out 管理。
- 战役时间线。

验收标准：

- KP 可以在一个页面看到房间运行状态。
- KP 可以快速打开任意玩家角色卡。
- KP 可以修改角色卡并保存。

### 3.8 界面背景自定义

必须支持：

- 默认界面背景为纯黑。
- 用户可以从预设纯色背景中切换界面背景。
- 预设背景至少包含：纯黑、深灰、墨绿、深蓝、暗红。
- 背景设置应优先作为本地个人偏好保存，不影响其他玩家。
- 后续可以扩展为房间级主题，由 KP 设置并同步给所有玩家。

原则：

- 第一版只做纯色背景，不做复杂图片、动态背景或大面积渐变。
- 背景切换不能影响文字可读性。
- 所有预设背景下，主要按钮、输入框、聊天内容和角色卡文字都必须保持足够对比度。

验收标准：

- 新用户首次进入时看到纯黑背景。
- 用户可以在界面设置中切换到其他纯色背景。
- 刷新页面后，本地背景偏好仍然保留。
- 不同用户可以拥有不同的本地背景设置。

## 4. 非功能需求

### 准确性

- 规则查询必须来源可追溯。
- 骰子结果必须可信。
- 角色卡解析失败时要明确提示，而不是静默丢字段。

### 可维护性

- 前后端类型尽量统一。
- 角色卡解析逻辑必须独立成模块。
- 骰子逻辑必须独立成模块。
- 规则书索引必须可以重新生成。

### 隐私与版权

- 规则书 PDF 只作为用户本地或私有部署资料。
- 不把完整规则书文本作为公开接口暴露。
- 录音和聊天记录默认只对房间成员可见。

### 可扩展性

- 后续可以接入实时语音。
- 后续可以接入 AI 转写和总结。
- 后续可以支持多种角色卡模板。
- 后续可以支持多个 COC 规则版本。

## 5. 第一版优先级

第一版优先做：

- 房间。
- 文字聊天。
- 默认纯黑背景与纯色背景切换。
- 角色卡上传解析。
- KP 查看和修改角色卡。
- 骰子和快捷检定。
- 投掷日志。
- 本地规则书检索。

第一版可以暂缓：

- 完整多人实时语音。
- 自动语音转写。
- 完整 AI 总结。
- NPC 管理。
- 地图和手out。
- 多规则书版本管理。

## 6. 成功标准

第一版成功标准：

- KP 能创建一个房间。
- 玩家能进入房间并上传角色卡。
- 系统能自动展示角色卡主要属性和技能。
- 玩家能点击属性或技能进行 COC 检定。
- 投掷结果能同步到聊天并保存。
- KP 能查看并修改玩家角色卡。
- 用户能搜索本地规则书并看到页码来源。

最终版本成功标准：

- 一个 COC 小队可以完整使用该系统完成一次线上跑团。
- 跑团过程中的聊天、语音、骰子、角色卡变化和规则引用都能被保存。
- 结束后可以生成可信的跑团总结。
- 规则查询足够准确，不依赖模型幻觉。
